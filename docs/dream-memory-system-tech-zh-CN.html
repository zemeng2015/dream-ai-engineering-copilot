DREAM 记忆系统技术说明

本文是 DREAM 记忆系统中文说明，覆盖设计原理、四层实现、技术栈、验收机制和下一步优化空间。

1. 项目定位

DREAM 是一个 source-backed memory platform。它不是通用聊天机器人，也不是简单 RAG demo。核心目标是把团队里的代码、文档、runbook、事故记录、历史 Jira/PR、测试计划、评审规则等信息，整理成可检索、可引用、可审计、可人工验收的工程记忆系统。

2. 设计原则

  • 证据优先：任何可持久化的记忆都必须能回溯到 source、span、hash 或具体文件路径。
  • 少提取、强引用：优先提取低风险、可验证、可审查的 claim。
  • 结构化记忆优先于 prompt 拼接：团队知识沉淀为索引、图谱、claim 和审计记录。
  • 语义 claim 默认需要人工验收：代码结构类事实可以自动批准；文档推断类记忆默认 candidate。
  • 记忆不能污染自己：生成物不能直接变成权威记忆，除非能指向原始 source 并通过 review/eval。

3. 高层架构

Knowledge Packs / Repo / Runbooks / Incidents / PR-Jira History
  -> Source Registry
  -> Source Spans
  -> Codebase Index
  -> Evidence Graph
  -> Candidate Memory Claims
  -> Validation Gates
  -> Memory Diff Review
  -> Durable Memory Ledger
  -> Retrieval for Requirement / PR Review / Eval Workflows

4. 记忆系统的四层实现

4.1 Knowledge Pack 层

Knowledge Pack 是团队知识源头，由 Markdown 和 YAML 组成，包含 domain docs、architecture docs、runbooks、incidents、testing docs、pr-review rules、historical Jira/PR 和 concept memory。

实现组件：KnowledgePackLoader、MarkdownDocumentLoader、Chunker、SimpleRetriever。

4.2 Codebase Memory 层

Codebase Memory 把本地 repo 变成结构化代码记忆，记录 files、language、file role、symbols、dependencies、concepts、source-to-test mappings、summaries 和 warnings。

artifact 路径：

artifacts/codebase-indexes/{team_id}/{repo_name}.json

4.3 Evidence Graph 层

Evidence Graph Lite 把文档、代码、测试、事故、历史 Jira/PR 连接成图，边包括 MENTIONED_IN、IMPLEMENTED_BY、DEFINED_IN、TESTED_BY、AFFECTS、REGRESSED_BY、REQUIRED_BY 和 CHANGED_BY。

artifact 路径：

artifacts/evidence-graphs/{team_id}/{repo_name}.json

4.4 Governed Memory Distillation 层

记忆蒸馏层把 repo 和知识文档转成可审查的 MemoryClaim。每条 claim 包含 source span、hash、提取方法、置信度、治理状态、安全分类和审计时间。每次 scan 也会记录 schema version、scanned repo path、Git root、commit SHA、dirty state、dirty path list 和 scanner version。

SourceRecord.commit_sha 会从 scan provenance 填充。hash 仍基于原始内容计算，但 SourceSpan.preview 在落盘前会先 redaction，避免 secret-like assignment、AWS access key、JWT-like token、private key header 进入 artifact preview。

artifact 路径：

artifacts/memory-scans/{team_id}/{scan_id}.json
artifacts/memory-scans/{team_id}/latest.json
artifacts/memory-evals/{team_id}/{evaluation_id}.json

5. Claim 类型和晋升规则

Claim 类型当前策略
file has languagedeterministic，自动 approved
symbol defined in fileparser/indexer 提取，自动 approved
endpoint-like symbol直接代码证据存在时 approved
source-to-test mapping默认 candidate，后续需要更强证据
concept/runbook/incident/decision memorysemantic candidate，需要人工 review
secret-like sourcequarantined

6. 验收和 Guardrails

dream memory eval 检查 citation validity、unsupported claim rate、secret leakage count、structural claim count、semantic candidate claim count 和 auto-promoted semantic claim count。

citation_validity == 1.0
unsupported_claim_rate <= 0.03
secret_leakage_count == 0
auto_promoted_semantic_claims == 0

8. 技术栈

  • 后端：Python 3.11+、FastAPI、Pydantic v2、Typer CLI、PyYAML、Uvicorn、SQLite、JSON artifacts。
  • 开发测试：pytest、ruff、httpx。
  • 前端：Angular 19、TypeScript 5.7、RxJS、SCSS、Karma/Jasmine。
  • 当前 memory/index/graph 层不依赖外部数据库、向量数据库或云服务。

9. 当前实现的强项

  • 本地可运行，部署和演示成本低。
  • 证据链清楚：source、span、hash、artifact 都能落盘。
  • 结构化程度高：形成 codebase index、evidence graph、memory claim，而不是直接拼 prompt。
  • Governance 意识正确：semantic claim 默认 candidate。
  • API/CLI 都已经可用，适合继续接 UI review queue。

10. 主要优化空间

  1. 已完成基础版：记录 commit SHA、dirty state、scanned root、extractor version、artifact schema version。
  2. P0：实现 durable approval ledger，支持 approve/reject/quarantine、reviewer、reason、superseded_by。
  3. 已完成基础版：secret redaction，在写入 preview 前先脱敏。后续可继续增强高熵 token 和 .env 语义检测。
  4. P1：实现真正的跨 scan diff，识别 added、removed、changed、stale evidence。
  5. P1：新增 MemoryClaimRetriever，只让 approved claims 进入业务检索。
  6. P1：引入 schema-based LLM extraction，但 semantic claim 仍默认 candidate。
  7. P2：按 content hash 做增量扫描。

11. 建议下一步开发顺序

  1. 提交当前 memory distillation MVP。
  2. 加 approval ledger。
  3. 加 true scan diff。
  4. 加 MemoryClaimRetriever。
  5. 继续增强 secret scanning。
  6. 最后再加 LLM semantic extraction。
